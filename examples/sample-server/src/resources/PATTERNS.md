# Three ways to structure a use case in a Rockets app

This sample-server **deliberately** uses three different patterns side by side
so you can read them, compare them, and pick what fits each feature you build.
None is "more correct" than the others; they trade boilerplate, flexibility,
and team familiarity differently.

| | **Pattern C — Declarative `defineResource`** | **Pattern A — Service + Controller** | **Pattern B — CQRS (Command + Handler)** |
|---|---|---|---|
| **Used by** | `pet/`, `pet-vaccination/`, `tag/`, `appointment/` (+ `reminder/`) | `pet-share/`, `admin/`, `audit/` | `pet-transfer/` |
| **Routes** | Auto-generated by core from `operations[]` | Hand-written `@Controller` | Hand-written `@Controller` that dispatches via `CommandBus` |
| **Business logic lives in** | A `@CommandHandler(CrudCreateCommand)` (etc.) class — only when you need to override default CRUD | An `@Injectable()` service method | A `@CommandHandler(MyVerb)` class |
| **Boilerplate** | Lowest (the framework writes the controller for you) | Low (one service + one controller) | Highest (Command class + Handler class + Module wiring + Controller) |
| **Discoverability** | Search `defineResource` for the entity | Search `@Controller(...)` for the path | Search `new XCommand(...)` for every call site |
| **Reusable from non-HTTP entry points (CLI, queue, cron)?** | No — generated controllers are HTTP-only; the handlers are CRUD-shaped | Yes — call the service directly | Yes — `commandBus.execute(new XCommand(...))` from anywhere |
| **Side effects via `EventBus`?** | Yes (handlers can publish) | Yes (services can publish) | Yes — and the Command class makes the *trigger* discoverable |

## All three share the same persistence contract

Regardless of wrapping, every use case in this repo follows the same three
rules. The persistence stack
(`@concepta/nestjs-repository` + `TransactionScope` + `TrxCtx` overlay) is
**orchestration-style-agnostic** — CQRS isn't required, it's one way to package
the orchestration.

1. **Accept `ctx` at the entry point** — from `getAppContext(request)` on HTTP
   routes, or from `command.context` on custom CRUD command/query handlers
   (`CrudContextInterface`).
2. **Wrap multi-step work in `TransactionScope.run(ctx, txCtx => ...)`** so the
   read + write share one snapshot.
3. **Call repositories with `{ ctx: txCtx }`** so they enlist in the active
   transaction.

### Request-context types (`AppContextInterface` vs `CrudContextInterface`)

The object you thread into services and `{ ctx }` on repository calls is
**not always the same TypeScript type**, even when the runtime value is the
same host:

| Entry point | Source | Type to use |
|---|---|---|
| Hand-written controllers, services, hand-written CQRS commands built from `getAppContext(req)` | `getAppContext(request)` from `@concepta/nestjs-common` | `AppContextInterface` |
| Custom CRUD command/query handlers (`PetCreateHandler`, `PetUpdateHandler`, …) | `command.context` on `CrudCreateCommand` / `CrudUpdateCommand` / … | `CrudContextInterface<…>` from `@bitwild/rockets-crud` |

`AppContextInterface` is the overlay API (`require`, `with`, `defineOverlay`).
`CrudUpdateCommand.context` is typed as `CrudContextInterface` and does **not**
extend `AppContextInterface` in the type system, so handlers that receive
`command.context` should type parameters as `CrudContextInterface`. Both remain
`PlainLiteralObject`, so `repo.findOne({ …, ctx: context })` stays valid for
either.

## Pattern C — Declarative `defineResource()`  (default for CRUD-shaped resources)

**Folder shape (`resources/pet/`):**

```
pet/
  pet.entity.ts            ← TypeORM entity
  pet.dto.ts               ← Create / Update / Response DTOs
  pet.resource.ts          ← defineResource() bundle: routes, operations, hooks
  pet.constants.ts         ← entity key, table name, etc.
  pet-create.handler.ts    ← @CommandHandler(CrudCreateCommand) — only for the
  pet-update.handler.ts      operations whose default behaviour you override
  pet-tag.entity.ts        ← M2M junction (no own resource)
  pet-tag.syncer.ts        ← extra provider used by the handlers
  index.ts
```

**Wiring:** `app.module.ts` calls `RocketsModule.forRootAsync({ resources: [petResource, …] })`.
The framework builds the controller, the validation pipeline, the Swagger
metadata, and dispatches each operation to either its default handler or the
custom one declared in `operations[].commandHandler`.

**Pick this when:**

- The endpoint set is the standard list/read/create/update/delete (or
  soft-delete/restore) for a single aggregate.
- You only need to customise *some* operations (e.g. owner-stamping on create,
  many-to-many syncing on update). Default operations stay default.
- You want hooks (`OwnerScopeHook`, `AuditLogHook`, etc.) to compose
  declaratively.

**Skip this when:**

- The endpoint isn't CRUD-shaped (e.g. `POST /pets/:id/share` is a verb on a
  pet, not a CRUD op on a Share — though that's also a fine choice).
- You need to reach the use case from a non-HTTP dispatcher (CLI, queue).

### Sub-flavour: extracted `commands/` + `events/` folders

Even with `defineResource()`, when a single resource grows multiple custom
command handlers and domain events you can extract them into per-folder
subdirectories. `pet-transfer/` shows the fully extracted layout
(`commands/impl/`, `commands/handlers/`, `events/`, `dto/`); `pet/` shows the
flat layout (handlers right next to entity/dto). Both are correct — pick
whichever stays readable for your team.

## Pattern A — Plain service + dedicated controller (`pet-share/`, `admin/`, `audit/`)

**Folder shape (`resources/pet-share/`):**

```
pet-share/
  pet-share.entity.ts      ← TypeORM entity (junction)
  pet-share.dto.ts         ← request / response DTOs
  pet-share.controller.ts  ← @Controller, hand-written routes
  pet-share.service.ts     ← @Injectable, business logic + repo calls
  pet-share.feature.ts     ← defineModuleResource({ entities, module })
  pet-share.constants.ts
  pet-owner-or-shared.hook.ts  ← repo hook reused by /pets queries
  index.ts
```

The `pet-share.feature.ts` bundle owns both the `PetShareEntity` row
(registered with the root `repository` adapter) and the Nest slice
(`controllers`, `providers`, `exports`). It's listed in `app.module`'s
`RocketsModule.forRootAsync({ resources: [...] })` alongside CRUD bundles
— no separate `@Module` file or `repositories.entities[]` block required.

```ts
// controller
@Post()
share(@Req() req: Request, @Body() dto: PetShareCreateDto) {
  return this.petShareService.share(getAppContext(req), { ... });
}

// service
@Injectable()
export class PetShareService {
  constructor(
    private readonly txScope: TransactionScope,
    @InjectDynamicRepository(PetShareEntity)
    private readonly shareRepo: RepositoryInterface<PetShareEntity>,
  ) {}

  async share(ctx: AppContextInterface, input: ShareInput) {
    return this.txScope.run(ctx, async (txCtx) => {
      await this.requireOwnedPet(txCtx, input.petId, input.actorUserId);
      return this.shareRepo.create({ ... }, { ctx: txCtx });
    });
  }
}
```

**Pick this when:**

- The operation is a small CRUD-ish write or read with one or two steps.
- You don't (yet) need other parts of the system to react to the outcome.
- You want minimum boilerplate (no Command class, no `@CommandHandler`).
- The route doesn't fit a CRUD shape (`POST /pets/:petId/share`) and you don't
  want to bend `defineResource()` to match.

## Pattern B — CQRS Command + Handler (`pet-transfer/`)

**Folder shape (`resources/pet-transfer/`):**

```
pet-transfer/
  pet-transfer.controller.ts        ← thin: @Controller + commandBus.execute
  pet-transfer.feature.ts           ← defineModuleResource({ entities: [], module })
  commands/
    impl/
      transfer-pet-ownership.command.ts   ← pure data carrier
    handlers/
      transfer-pet-ownership.handler.ts   ← @CommandHandler(TransferPetOwnershipCommand)
  events/
    pet-transferred.event.ts        ← published from the handler
  dto/
    transfer-pet-ownership.dto.ts   ← request body
  index.ts
```

`pet-transfer` consumes the pet repository owned by `petResource` — no
new tables — so `petTransferFeature` uses `entities: []`. The bundle still
contributes its `CqrsModule` import, controller, and handler through the
`module` slice.

```ts
// controller
@Post()
transfer(@Req() req: Request, @Param('petId') petId: string, @Body() dto: TransferPetOwnershipDto) {
  return this.commandBus.execute(
    new TransferPetOwnershipCommand(getAppContext(req), petId, /* actor */ user.id, dto.newOwnerId),
  );
}

// command (pure data, no logic)
export class TransferPetOwnershipCommand {
  constructor(
    public readonly ctx: AppContextInterface,
    public readonly petId: string,
    public readonly actorUserId: string,
    public readonly newOwnerId: string,
  ) {}
}

// handler
@CommandHandler(TransferPetOwnershipCommand)
export class TransferPetOwnershipHandler implements ICommandHandler<TransferPetOwnershipCommand> {
  async execute(cmd: TransferPetOwnershipCommand) {
    return this.txScope.run(cmd.ctx, async (txCtx) => {
      // ...verify ownership, update petRepo, publish event
      this.eventBus.publish(new PetTransferredEvent(cmd.petId, cmd.actorUserId, cmd.newOwnerId));
    });
  }
}
```

**Pick this when:**

- The operation is a **business verb** (Transfer, Approve, Cancel, Publish)
  with non-trivial invariants — the Command class + handler pairing makes the
  use case discoverable (grep the command name, find every call site and the
  one place it executes).
- You want **listeners** to react via `EventBus.publish(new XEvent(...))`
  without the handler knowing about them — audit trails, notifications, cache
  invalidation.
- The operation might be **composed** inside another handler via
  `commandBus.execute(new SubCommand(...))` (typed sub-workflow).
- You want the use case reachable from a CLI / message queue / scheduled job
  without dragging HTTP concerns along — the command is the stable contract;
  the controller is one of many possible dispatchers.

## How to migrate between patterns

The persistence code (`txScope.run`, repo calls with `{ ctx: txCtx }`)
**doesn't change** when you switch packaging styles. The switch is mechanical:

- **Pattern A → B (Service → CQRS):** wrap the service-method body in a
  `@CommandHandler(NewCommand)` class, move parameters into the Command, swap
  the controller to `commandBus.execute(new NewCommand(...))`. Done.
- **Pattern B → A (CQRS → Service):** inline the handler body into a service
  method, delete the Command class, have the controller call the service
  directly. Done.
- **Pattern A or B → C (Service/CQRS → defineResource):** only worth it if
  your hand-written routes happen to be standard CRUD on a single entity.
  Replace the controller + service/handler with a `defineResource({ entity,
  operations: [...] })` bundle and let the framework generate the routes;
  custom mutations move into per-operation `commandHandler`s.

Pick what fits each use case today; promote to a different style when the
requirements change.
