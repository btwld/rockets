# `sample-server`

End-to-end NestJS app using **[@bitwild/rockets](../../packages/rockets-server/)**
with declarative resources, dynamic repositories, and **three
side-by-side orchestration styles**.

> Read this as a *menu* of patterns, not a mandatory template. Different
> resources here intentionally use different patterns so you can compare
> them directly.

## Run it

```bash
# from repo root
yarn workspace sample-server install
yarn workspace sample-server start:dev
```

- HTTP: `http://localhost:3000`
- Swagger: `http://localhost:3000/api`

```bash
yarn workspace sample-server build
yarn workspace sample-server test:e2e
```

E2E lives in `test/sample-server.e2e-spec.ts` and is the primary
behaviour spec (auth, `/me`, CRUD, sharing, admin, appointments, audit,
events, transfers).

A QA test plan for Playwright is in [`QA_TEST_CASES.md`](./QA_TEST_CASES.md).

## How this app is wired

`AppModule` composes:

1. **TypeORM (SQLite in-memory)** — demo database only.
2. **`AuthModule`** — local JWT signup/login + `SampleAuthAdapter` for
   Rockets. In a real external-IdP app, replace with your provider and
   keep the `RocketsModule` wiring.
3. **`RocketsModule.forRootAsync`** — external-auth-style integration:
   `authProvider`, `userMetadata` DTOs/entity, top-level `repository`
   (`TypeOrmRepositoryModule`), and `resources[]` mixing `defineResource()`
   bundles with `defineModuleResource()` bundles.

Standalone entities used only through dynamic repos (not owned by a
`defineResource` bundle) are registered via `defineModuleResource({ entities:
[...], module: { ... } })` — e.g. `userReposFeature`, `petShareFeature`,
`auditFeature`. CQRS-only flows like `petTransferFeature` use `entities: []`.

## The three patterns at a glance

Deep dive (with code samples and decision criteria):
[`src/resources/PATTERNS.md`](./src/resources/PATTERNS.md).

|   | **C — Declarative `defineResource`** | **A — Service + Controller** | **B — CQRS** |
|---|---|---|---|
| **Used by** | `pet/`, `pet-vaccination/`, `tag/`, `appointment/` | `pet-share/`, `admin/`, `audit/` | `pet-transfer/` |
| **Routes** | Auto-generated by core | Hand-written `@Controller` | Hand-written `@Controller` → `commandBus.execute` |
| **Logic in** | `@CommandHandler(CrudCreateCommand)` etc. (only for overrides) | `@Injectable()` service method | `@CommandHandler(MyVerb)` class |
| **Boilerplate** | Lowest | Low | Highest |
| **Reusable from CLI / queue / cron?** | No (HTTP-only) | Yes | Yes (Command IS the contract) |
| **Best for** | Standard CRUD on a single entity | Small custom routes that don't fit CRUD | Business *verbs* with invariants and side effects |

All three share the same persistence contract: `getAppContext(req)` →
`TransactionScope.run` → `repo.method({ ctx })`. Switching between
styles is mechanical.

## Module / resource map

| Area | Location | Pattern | What to learn |
|---|---|---|---|
| **Auth** | `src/auth/` | own (signup/login + JWT) | Minimal built-in auth so the sample mints tokens without an external IdP |
| **Pet** | `src/resources/pet/` | C — `defineResource` | Full template: DTOs, custom Create/Update handlers, hooks, relations to vaccination + tags, soft-delete + restore, M2M sync |
| **Pet vaccination** | `src/resources/pet-vaccination/` | C — `defineResource` | Child resource keyed for relation targets |
| **Tag** | `src/resources/tag/` | C — `defineResource` | Small resource used for M2M with pets |
| **Appointment + reminder** | `src/resources/appointment/` | C ×2 | Nested transactional flow + separate resource |
| **Pet share** | `src/resources/pet-share/` | A — Service + Controller | Hand-written routes when CRUD shape doesn't fit |
| **Pet transfer** | `src/resources/pet-transfer/` | B — CQRS | Same persistence rules, Command + Handler + EventBus packaging |
| **Admin** | `src/admin/` | A — Service + Controller | Guarded admin surface (`AdminGuard`) calling domain services |
| **Audit** | `src/audit/` | A — Service + Controller | Cross-cutting audit entity + hook participation |
| **Events** | `src/events/` | infrastructure | Async side effects via `@nestjs/cqrs` events |

## How to choose for *your* feature

```
Is the endpoint set standard CRUD on a single entity?
├── Yes  → Pattern C (defineResource).
│         Override individual ops with commandHandler when needed.
│         Use hooks for owner-scoping, audit, etc.
│
└── No   → Is the operation a business verb with invariants
          (Transfer, Approve, Publish, Cancel)?
          ├── Yes → Pattern B (CQRS). Command class is discoverable.
          │
          └── No  → Pattern A (service + controller).
                    Cheapest custom route. Promote to B if you later
                    need typed sub-commands or non-HTTP dispatch.
```

> The patterns are not a hierarchy. A medium-sized app mixes all three.
> Migration cost between A and B is minutes (see PATTERNS.md).
> C → A or C → B happens when standard CRUD stops being enough.

## Layout differences

```
resources/pet/                       ← Pattern C, flat
  pet.entity.ts
  pet.dto.ts
  pet.resource.ts                    ← defineResource() call
  pet-create.handler.ts              ← override Create only
  pet-update.handler.ts              ← override Update only
  pet-tag.entity.ts                  ← M2M junction
  pet-tag.syncer.ts                  ← extra provider
  index.ts

resources/pet-share/                 ← Pattern A
  pet-share.entity.ts
  pet-share.dto.ts
  pet-share.controller.ts
  pet-share.service.ts
  pet-share.module.ts
  pet-owner-or-shared.hook.ts
  index.ts

resources/pet-transfer/              ← Pattern B
  pet-transfer.controller.ts
  pet-transfer.module.ts
  commands/
    impl/
      transfer-pet-ownership.command.ts
    handlers/
      transfer-pet-ownership.handler.ts
  events/
    pet-transferred.event.ts
  dto/
    transfer-pet-ownership.dto.ts
  index.ts
```
