# Sample Server (`examples/sample-server`)

Reference NestJS app that exercises **[@bitwild/rockets](https://www.npmjs.com/package/@bitwild/rockets)** (Rockets Server) together with **declarative resources** (`defineResource`), **dynamic repositories**, and **three side-by-side orchestration styles**.

> Read this as a *menu* of patterns, not as a mandatory template. Different resources here use **different patterns on purpose** so you can see them side by side and copy whichever fits each feature you build.

---

## Run

```bash
# from repo root
yarn workspace sample-server install
yarn workspace sample-server start:dev
```

- HTTP: `http://localhost:3000`
- Swagger: `http://localhost:3000/api` (registered via `SwaggerUiService` in `src/main.ts`)

```bash
yarn workspace sample-server build
yarn workspace sample-server test:e2e
```

E2E lives in `test/sample-server.e2e-spec.ts` and is the primary specification for behaviour (auth, `/me`, CRUD, sharing, admin, appointments, audit, events, transfers).

A QA test plan for Playwright is in [`QA_TEST_CASES.md`](./QA_TEST_CASES.md).

## How this app is wired

`AppModule` composes:

1. **TypeORM (SQLite in-memory)** — demo database only.
2. **`AuthModule`** — local JWT signup/login + `SampleAuthProvider` for Rockets.
3. **`RocketsModule.forRootAsync`** — external-auth-style integration: `authProvider`, `userMetadata` DTOs/entity, unified `repositories` (`TypeOrmRepositoryModule` + extra entity keys), and `resources[]` built from `defineResource()` factories.

Standalone entities used only through dynamic repos (not owned by a `defineResource` bundle) are listed under `repositories.entities` (user, pet-share junction, audit log).

## The three patterns at a glance

Deep dive with code samples and decision criteria: [`src/resources/PATTERNS.md`](./src/resources/PATTERNS.md).

| | **Pattern C — Declarative `defineResource`** | **Pattern A — Service + Controller** | **Pattern B — CQRS** |
|---|---|---|---|
| **Used by** | `pet/`, `pet-vaccination/`, `tag/`, `appointment/` (incl. reminder) | `pet-share/`, `admin/`, `audit/` | `pet-transfer/` |
| **Routes** | Auto-generated by core | Hand-written `@Controller` | Hand-written `@Controller` → `commandBus.execute` |
| **Logic lives in** | `@CommandHandler(CrudCreateCommand)` etc. (only when overriding default CRUD) | `@Injectable()` service method | `@CommandHandler(MyVerb)` class |
| **Boilerplate** | Lowest | Low | Highest |
| **Reusable from CLI / queue / cron?** | No (HTTP-only) | Yes | Yes (the Command is the contract) |
| **Best for** | Standard CRUD on a single entity | Small custom routes that don't fit CRUD shape | Business *verbs* with invariants and side effects |

All three share the same persistence contract (`getAppContext(req)` → `TransactionScope.run` → `repo.method({ ctx })`). Switching styles is mechanical — see PATTERNS.md.

## Module / resource map — what each area teaches

| Area | Location | Pattern | What to learn |
|---|---|---|---|
| **Auth** | `src/auth/` | own (signup/login + JWT) | Minimal **built-in-style** auth alongside Rockets so the sample can mint tokens without an external IdP. In a real external-IdP app, replace this with your provider and keep the `RocketsModule` wiring. |
| **Pet** | `src/resources/pet/` | **C — `defineResource`** | The full template: DTOs, custom **create/update** command handlers, **hooks** (`PetOwnerOrSharedHook`, `AuditLogHook`), **relations** to vaccination + tags, **soft-delete + restore** with per-operation **response** overrides, and `PetTagSyncer` as an extra provider for many-to-many sync. **Flat layout** (handlers next to entity/dto). |
| **Pet vaccination** | `src/resources/pet-vaccination/` | **C — `defineResource`** | Child resource keyed for `aggregateResources` relation targets from `pet`. CRUD scoped as a separate resource. |
| **Tag** | `src/resources/tag/` | **C — `defineResource`** | Small resource used for **many-to-many** with pets; relation target from `pet.resource.ts`. |
| **Pet-tag junction** | `src/resources/pet/pet-tag.entity.ts` | persistence only | M2M table; registered in `repositories.entities` because it is not the primary entity of its own `defineResource` bundle. |
| **Appointment + reminder** | `src/resources/appointment/` | **C — `defineResource` ×2** | **Nested transactional** flow (appointment + reminder created together) via custom handler, plus a **separate `defineResource`** for reminders. Demonstrates multi-entity coordination inside the resource model. |
| **Pet share** | `src/resources/pet-share/` | **A — Service + Controller** | Hand-written routes (`POST /pets/:petId/share`) backed by a service with `TransactionScope` and dynamic repo. Shows "Rockets + bespoke HTTP" when CRUD shape doesn't fit. |
| **Pet transfer** | `src/resources/pet-transfer/` | **B — CQRS** | Same persistence rules, different packaging: `Command` + `@CommandHandler` + `EventBus`. **Extracted layout** (`commands/impl/`, `commands/handlers/`, `events/`, `dto/`) so big CQRS modules stay navigable. |
| **Admin** | `src/admin/` | **A — Service + Controller** | **Guarded admin surface** (`AdminGuard`) calling into domain services; illustrates elevated routes outside CRUD resources. |
| **Audit** | `src/audit/` | **A — Service + Controller** | **Cross-cutting audit** entity + hook (`AuditLogHook`) participation from CRUD controllers. |
| **Events** | `src/events/` | infrastructure | **Nest events** (`@nestjs/cqrs` event layer): listener reacts to `PetCreatedEvent`; `FakeEmailGateway` for tests. Shows async side effects decoupled from HTTP handlers. |

## How to choose for *your* feature

Quick decision tree:

```
Is the endpoint set standard CRUD on a single entity?
├── Yes  → Pattern C (defineResource).
│         Override individual operations with commandHandler when defaults
│         aren't enough. Use hooks for owner-scoping, audit, etc.
│
└── No   → Is the operation a business verb with invariants
          (Transfer, Approve, Publish, Cancel)?
          ├── Yes → Pattern B (CQRS). Command class makes it discoverable.
          │
          └── No  → Pattern A (service + controller).
                    Cheapest custom route. Promote to B if you need
                    typed sub-commands or non-HTTP dispatch later.
```

> The patterns are not a hierarchy. A medium-sized app will mix all three. The migration cost between A and B is minutes (see PATTERNS.md "How to migrate"); promoting C → A or C → B happens when standard CRUD stops being enough.

## Layout differences you can see in the file tree

Different patterns produce visibly different folder shapes — this is on purpose so the trade-off is concrete:

```
resources/pet/                       ← Pattern C, flat layout
  pet.entity.ts
  pet.dto.ts
  pet.resource.ts                    ← defineResource() call
  pet.constants.ts
  pet-create.handler.ts              ← override Create only
  pet-update.handler.ts              ← override Update only
  pet-tag.entity.ts                  ← M2M junction
  pet-tag.syncer.ts                  ← extra provider used by handlers
  index.ts

resources/pet-share/                 ← Pattern A, service+controller
  pet-share.entity.ts
  pet-share.dto.ts
  pet-share.controller.ts
  pet-share.service.ts
  pet-share.module.ts
  pet-owner-or-shared.hook.ts
  pet-share.constants.ts
  index.ts

resources/pet-transfer/              ← Pattern B, extracted CQRS
  pet-transfer.controller.ts
  pet-transfer.module.ts
  commands/
    impl/
      transfer-pet-ownership.command.ts
    handlers/
      transfer-pet-ownership.handler.ts
  events/
    pet-transferred.event.ts
  dto/
    transfer-pet-ownership.dto.ts
  index.ts
```
